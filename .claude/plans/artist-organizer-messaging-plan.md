# Implementation Plan: Q&A messaging between artists and organizers

Spec: `.claude/plans/artist-organizer-messaging-spec.md`

## Technical Decisions

- **Schema: two tables.** `event_threads` (one row per artist × event, unique (event_id, artist_profile_id), carries per-side `last_read_at`) + `event_thread_messages` (append-only, author + body + timestamp). Chosen over a single-table design so read-state has an unambiguous home and the organizer inbox is a single indexed scan.
- **Two notification types, not one.** `thread_message_from_artist` (recipient = organizer) and `thread_message_from_organizer` (recipient = artist). Matches the per-direction precedent (`new_application` vs `application_revoked`), keeps preferences independent, yields distinct subjects and deep-links.
- **"Also post as announcement" is a single server-side fan-out.** Organizer reply action inserts the thread message, fires the private notification, then conditionally inserts into `event_announcements` and calls the existing `notifyAcceptedApplicants` path. No schema overlap between threads and announcements.
- **No thread subject.** One timeline per (artist, event). Spec explicit.
- **Plain text only, cap 5000 chars.** `whitespace-pre-line` rendering to match announcements. No Markdown this iteration.
- **Organizer inbox: same page + dialog.** New "Questions from artists" card on the event management page, sibling to the Announcements card. Clicking a thread row opens a `<Dialog>` (same primitive as publish confirmation) containing the timeline + reply composer.
- **Artist UI: nested inside existing Results card.** "Chat with the organizer" block lives at the bottom of the post-publish `SectionCard` (gated on `ownApplicationStatus === "accepted"`).
- **Revalidation pattern mirrors announcements** — every mutating action ends with `revalidatePath(\`/conventions/manage/events/${eventId}\`)` + `revalidatePath(\`/events/${eventId}\`)`.

## Tasks

### 1. Schema — `event_threads` + `event_thread_messages` ✅

Add the two tables and generate the Drizzle migration. No behavior change yet — this is pure infrastructure that unblocks every later task.

**Requirements:** Infrastructure (enables REQ-2, REQ-3, REQ-7).

**Files:**
- `src/lib/db/schema/event-threads.ts` — new. Two `pgTable` exports: `eventThreads` and `eventThreadMessages`. Column list per the decisions block above. Unique index `(event_id, artist_profile_id)` on `eventThreads`. Index on `(thread_id, created_at)` for `eventThreadMessages` (powers message-list queries). FKs: eventThreads.eventId → events.id (cascade), eventThreads.artistProfileId → profiles.id (cascade), eventThreadMessages.threadId → eventThreads.id (cascade), eventThreadMessages.authorProfileId → profiles.id (cascade).
- `src/lib/db/schema/index.ts` — (if an aggregator exists) export the new tables alongside existing ones.
- `src/lib/db/migrations/0016_<slug>.sql` — auto-generated by `npm run db:generate`.

**Approach:**
- Define tables in Drizzle per the shape above. Timestamps are `timestamp({ mode: "date" }).notNull().defaultNow()`; `lastReadAt` columns are nullable (no reads on first creation).
- Run `npm run db:generate` to produce the SQL migration; verify the generated file has two `CREATE TABLE`s, the unique index, and the secondary index.

**Verification:** `npm run db:generate` produces a migration; `npm run build` still succeeds (no TS errors); snapshot of `drizzle/schema.ts` includes both new tables.

**Depends on:** none.

### 4. Thread query helpers ✅

Pure read helpers used by both artist-facing and organizer-facing pages. No mutations, no auth — consumers handle that.

**Requirements:** Infrastructure (enables REQ-2, REQ-3, REQ-4, REQ-7).

**Files:**
- `src/lib/threads/queries.ts` — new. Three helpers:
  - `getThreadForArtist(eventId, artistProfileId)` → `{ thread, messages } | null`. Returns null if no thread exists yet (artist hasn't asked anything).
  - `getThreadsForOrganizer(eventId)` → list of `{ thread, artist: {profileId, displayName, avatarUrl?}, lastMessagePreview, unreadForOrganizer }[]` sorted by `last_message_at` desc. `unreadForOrganizer` = `lastMessageAt > organizerLastReadAt || organizerLastReadAt === null` AND last message author is the artist.
  - `getThreadByIdForOrganizer(eventId, threadId)` → full thread + all messages + artist identity; used by the dialog.
- `__tests__/integration/thread-queries.test.ts` — asserts sorting order, unread flag, empty state.

**Approach:** Single Drizzle JOIN per helper; messages via secondary query ordered `asc(createdAt)`.

**Verification:** Query tests pass.

**Depends on:** 1.

### 5. Artist-facing server actions ✅

Post a message, mark the thread read. These are the actions the public event page mounts.

**Requirements:** REQ-1, REQ-2, REQ-7.

**Files:**
- `src/app/(public)/events/[eventId]/thread-actions.ts` — new. Two server actions with `"use server"` and the standard `(_prevState, formData): Promise<ActionState>` signature:
  - `sendThreadMessage` — Zod schema: `eventId` required, `body` min(1) max(5000). Auth: artist session + lookup the artist's `applications` row on `eventId`; require `status === "accepted"`. Upsert `eventThreads` on `(eventId, artistProfileId)` (insert-or-select; set `last_message_at = now()` on update). Insert `eventThreadMessages` with `authorProfileId = artistProfileId`. Look up the convention's `organizerId` and fire `notifyThreadMessageFromArtist`. Revalidate both paths.
  - `markThreadRead` — Artist session + own thread only. Update `eventThreads.artistLastReadAt = now()` where the thread belongs to this artist. Revalidate the public event page.
- `__tests__/integration/thread-messages-artist.test.ts` — cases: (a) accepted artist posts → thread + message exist + organizer notification fired; (b) waitlisted/rejected artist → error; (c) second message from same artist reuses the same thread row; (d) mark-as-read updates `artist_last_read_at`.

**Approach:**
- The "upsert" is implemented as `INSERT ... ON CONFLICT (event_id, artist_profile_id) DO UPDATE SET last_message_at = EXCLUDED.last_message_at RETURNING id` — single round-trip.

**Verification:** Integration test green.

**Depends on:** 1, 3, 4 (uses `getThreadForArtist`).

### 6. Organizer-facing server actions

Reply to a thread, optionally also post as announcement, mark the thread read.

**Requirements:** REQ-4, REQ-5, REQ-6, REQ-7.

**Files:**
- `src/app/(authenticated)/conventions/manage/events/[eventId]/thread-actions.ts` — new. Three actions:
  - `replyToThread` — Zod schema: `threadId`, `body` (min 1 max 5000), `alsoAsAnnouncement` (bool), `announcementSubject` (required iff `alsoAsAnnouncement=true`, min 1 max 200). Auth: organizer session + `getOrganizerEvent` ownership check + thread belongs to this event. Insert thread message with `authorProfileId = organizerProfileId`, bump `threads.last_message_at`. Fire `notifyThreadMessageFromOrganizer` to the artist. If `alsoAsAnnouncement=true`, insert `eventAnnouncements` row and call existing `notifyAcceptedApplicants(eventId, announcementSubject)`. Revalidate both paths.
  - `markThreadReadAsOrganizer` — organizer session + thread belongs to organizer's event. Update `eventThreads.organizerLastReadAt = now()`. Revalidate.
- `__tests__/integration/thread-messages-organizer.test.ts` — cases: (a) organizer replies → artist notification fired; (b) `alsoAsAnnouncement=true` creates `event_announcements` row and fires accepted-applicants notification; (c) non-owner organizer gets unauthorized; (d) mark-as-read updates `organizer_last_read_at`.

**Approach:** `alsoAsAnnouncement` runs in a `db.transaction` so the announcement and the thread message commit or roll back together.

**Verification:** Integration test green.

**Depends on:** 1, 3, 4.

- Mark-as-read: cleanest is a `useEffect` that fires a `fetch` to a tiny route handler, OR an invisible auto-submitting form. Given we already use form actions everywhere, use a client-side fetch POST to `/api/threads/{threadId}/read`.

Rely on the server-side read to happen when the artist next posts (which is when they're actively engaged). Spec's REQ-7 says "unread indicator until they open the thread" — we can satisfy this by marking read inside the page render (server action called from the page component when it detects messages and viewer is the artist). ## Risks

- **Enum-add migrations require postgres to accept `ALTER TYPE ADD VALUE` inside a transaction.** Drizzle generates these as separate statements (it's a known drizzle-kit behavior); if local dev DB is older than Postgres 12, the migration can fail. Mitigation: verify `drizzle-kit migrate` succeeds locally before committing task 2.
- **Thread upsert race.** Two rapid-fire messages from the same artist could both INSERT if they beat the unique index. Mitigated by `ON CONFLICT DO UPDATE RETURNING id` being atomic.
- **Read-state discipline.** We're marking threads read as a side effect of rendering (server-action call from the page). Spec-wise this is correct (the moment the user opens the thread the organizer expects them to have seen it), but it means refreshing the page clears any "unread" indicator even if the user didn't actually read the content. Acceptable for v1 given the target use case (small artist counts per event).
- **Bundle growth on the organizer page.** The inbox dialog pulls in the full thread content. For events with many artists this scales linearly; per-row dialogs are simplest but server queries grow linearly. If this bites, switch to a single dialog + client-side fetch of the selected thread. Not expected at current user scale.
- **Notification deep-links.** The artist notification links to `/events/${eventId}#thread` — relies on the event page mounting with the thread anchor rendered. The organizer link uses `#thread-${artistProfileId}`. Both must be covered by the jsdom tests so regressions are caught.
