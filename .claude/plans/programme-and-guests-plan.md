# Implementation Plan: Programme + Guests for events

Spec: `.claude/plans/programme-and-guests-spec.md`

## Technical Decisions

- **JSONB on `events`, flat arrays.** `events.programme: ProgrammeItem[] | null` and `events.guests: Guest[] | null`. Mirrors the existing `events.floorPlan` / `events.amenities` pattern. No new tables.
- **Whole-document save via server actions.** Two actions (`saveProgramme`, `saveGuests`) replace the full array on each call. Matches the floor-plan editor's `saveFloorPlan` precedent — simpler than per-item actions and reorders are a single save.
- **Guest image upload via REST.** New `POST /api/events/[eventId]/guests/image` mirroring `convention-logo` (multipart, magic-byte validation, `processImage` → webp, `storage.upload` to `events/<eventId>/guests/<guestId>.webp`). Returns the storage key; the editor stores it in the JSONB. The save action diffs prior-vs-new keys and deletes orphans — no separate DELETE endpoint.
- **Validation via Zod**, two new files (`lib/validations/programme.ts`, `lib/validations/guests.ts`). Cross-field rules (date-within-event-range, endTime > startTime) live alongside the field-level Zod schemas.
- **Organizer navigation:** add Programme + Guests as top-level tabs in `EventTabsNav` (revising the spec's earlier "no changes" line per the user). Tab order: Event details / Artists / Programme / Guests / Messaging / Floor plan.
- **Public tab order:** Overview / Programme / Guests / Artists / Floor plan / Practical info / Messages — per the design hand-off.
- **Public visibility:** both tabs gated on having any items. No publish flag, no auth gate. Renders for logged-out viewers.

## Tasks

### 1. Schema columns + event-context plumbing
Adds the two JSONB columns to `events`, generates the migration, threads the new fields through the shared event-viewer context loader.

**Requirements:** REQ-P1, REQ-G1 (foundation)

**Files:**
- `src/lib/db/schema/events.ts` — add `programme: jsonb("programme").$type<ProgrammeItem[]>()` and `guests: jsonb("guests").$type<Guest[]>()`. Define + export the `ProgrammeItem` and `Guest` interfaces alongside the existing `FloorPlan` types.
- `src/lib/db/migrations/00<n>_*.sql` — generated by `drizzle-kit generate`. Verify it's two additive `ADD COLUMN` statements, both nullable.
- `src/lib/events/event-context.ts` — add `programme` and `guests` to the `loadEventRow` select. Type-only ripple: `EventViewerContext.event.programme` / `.guests` become available downstream.

**Approach:**
- Insert the two columns next to `floorPlan` to keep view-related event data grouped.
- The interfaces:
  - `ProgrammeItem`: `{ id, date, startTime, title, endTime?, room?, speaker? }`
  - `Guest`: `{ id, name, title, role?, pronouns?, bio?, imagePath?, websiteUrl?, socialLinks?: SocialLink[] }`
  - Reuse `SocialLink` from `lib/artist-profile/social-links.ts`.
- Run `npm run db:generate` then `npm run db:migrate` + `npm run db:migrate:test`.
- No tests needed — schema-only, picked up by integration tests in later tasks.

**Verification:** `npm run db:migrate` clean. `npx tsc --noEmit` passes. `npm test` still 497 passing.

**Depends on:** none

### 3. Public Programme tab
New routed page `/events/[eventId]/programme` that groups items by date, sorts by startTime, renders the design hand-off's table-style schedule. Adds the tab to `ArtistEventTabsNav`.

**Requirements:** REQ-P3, REQ-P4, REQ-P6

**Files:**
- `src/app/(public)/events/[eventId]/programme/page.tsx` — new server component. Reads context, returns `notFound()` if `!event.programme || event.programme.length === 0`. Groups items by `date` (asc), sorts each group's items by `startTime` (asc, stable on insertion order). Renders a `<Card>` per day with a `<table>` inside (or stacked rows on small viewports).
- `src/components/events/artist-event-tabs-nav.tsx` — accept `showProgramme: boolean` prop. Insert the Programme tab at index 1 (between Overview and the existing Artists slot). Order: Overview / Programme / Guests / Artists / Floor plan / Practical info / Messages.
- `src/app/(public)/events/[eventId]/layout.tsx` — call `shouldShowProgrammeTab(ctx)` in the existing `Promise.all`, pass `showProgramme` to the tabs nav.

**Approach:**
- Day grouping logic: a small `groupBy` helper or inline reduce. Output `{ date: string; items: ProgrammeItem[] }[]` sorted by date.
- For each day card: header line (formatted with `formatDateNo`), then a list. Each row: time (mono) | title (semibold) | speaker (muted) | room as a `<Badge variant="secondary">` on the right.
- Mobile collapses the table into stacked rows (time + title on one line, speaker + room on the next).
- No edit affordance — public read-only.

**Verification:** Manually visit `/events/[id]/programme` with seed data. Tab appears in nav iff items exist. Direct URL → 404 when empty.

### 5. Programme save action + Zod validation
Server action `saveProgramme(prev, formData)` validates and writes the full array.

**Requirements:** REQ-P1, REQ-P2, REQ-P5

**Files:**
- `src/lib/validations/programme.ts` — Zod schemas: `programmeItemSchema` (id, date YYYY-MM-DD, startTime HH:mm, title 1–200, endTime?, room?<=80, speaker?<=200) and `programmeSchema = z.array(programmeItemSchema)`. Cross-field validators: `endTime > startTime` when both set; date inside event range (parameterised — pass `eventStart`, `eventEnd`).
- `src/app/(authenticated)/conventions/manage/events/[eventId]/programme/actions.ts` — new. `"use server"` server action mirroring `floor-plan/actions.ts`'s `authorize` helper. FormData carries a single `programme` field with the JSON-serialized array. Parses, validates against the Zod schema, writes via `db.update(events).set({ programme: parsed, updatedAt: new Date() }).where(eq(events.id, event.id))`. Returns `ActionState` with `fieldErrors` keyed on `items.<index>.<field>` when validation fails.
- `__tests__/integration/programme-save.test.ts` — new. Mirrors `floor-plan-publish.test.ts`:
  - Auth + ownership rejections
  - Valid array writes; invalid date / time / over-range rejects with field error
  - Empty array (clears programme to null or `[]`)
  - Cross-field: endTime ≤ startTime → fieldError on `endTime`
  - Cross-field: date outside event range → fieldError on `date`

**Approach:**
- Whole-document parse: `JSON.parse(formData.get("programme"))` wrapped in try/catch returning `{ error: "Invalid programme data" }` on parse failure.
- Validation surface: store `fieldErrors` keyed by `items.0.title`-style paths so the editor can map errors back to rows.
- Revalidation: organizer + public event paths (`/conventions/manage/events/<id>`, `/conventions/manage/events/<id>/programme`, `/events/<id>`, `/events/<id>/programme`).

**Verification:** `npm test -- programme-save` green. Type check clean.

**Depends on:** task 1

- Field-level error display: walk the `fieldErrors` map, surface inline beneath each affected field.

### 7. Guest image upload route
REST endpoint that accepts a multipart upload and returns a storage key. Mirrors the convention-logo route.

**Requirements:** REQ-G3 (foundation)

**Files:**
- `src/app/api/events/[eventId]/guests/image/route.ts` — new `POST` handler. Auth as organizer + ownership check via `getOrganizerEvent`. File validation (`File`, ≤ 5 MB, jpeg/png/webp via magic byte). `processImage` → webp. `storage.upload("events/${eventId}/guests/${guestId}.webp", processed.data, "image/webp")`. Returns `{ storagePath }` JSON. Body also requires a `guestId` field (UUID generated client-side) so we can address replacements.
- `__tests__/integration/guests-image-upload.test.ts` — new. Auth/ownership rejections, file-size + content-type rejections, happy path returns a key.

**Approach:**
- Reuse `processImage` and `storage` utilities — no new image-processing logic.
- The route does NOT delete prior images. That's the save action's job (diff old vs new keys).

**Verification:** `npm test -- guests-image-upload`. Manual upload via the Guests editor lands a file in storage and returns its key.

**Depends on:** task 1

### 8. Guests save action + Zod validation
Mirrors task 5 for guests, plus the orphan-image cleanup.

**Requirements:** REQ-G1, REQ-G2, REQ-G3 (deletion side)

**Files:**
- `src/lib/validations/guests.ts` — Zod schemas: `guestSchema` (id, name 1–200, title 1–80, role?<=200, pronouns?<=40, bio? markdown ≤ 4000, imagePath?, websiteUrl? URL, socialLinks?: array of `{ type, url }`). `guestsSchema = z.array(guestSchema)`.
- `src/app/(authenticated)/conventions/manage/events/[eventId]/guests/actions.ts` — new. Same pattern as programme action. Reads `formData.get("guests")` JSON, parses, validates, writes. Before write: load existing `event.guests`; compute the set of `imagePath` values previously stored vs new payload; for each removed image path matching the `events/<eventId>/guests/<id>.webp` shape, call `storage.delete(path).catch(() => {})`.
- `__tests__/integration/guests-save.test.ts` — new. Auth + ownership; valid array writes; field errors propagate; orphan-image cleanup verified by mocking `storage.delete` and checking it's called with the dropped path.

**Approach:**
- Defensive path-shape check before deleting: only delete keys under `events/<eventId>/guests/`. Prevents the action from reaching outside its sandbox if a malicious payload sneaks a different path.

**Verification:** `npm test -- guests-save`. Replacing an image path in the editor + saving → old key deleted, new one stays.

**Depends on:** tasks 1, 7

## Risks

- **`<Markdown>` server-vs-client.** If the existing `<Markdown>` component is client-only, the public Guests page must be marked `"use client"` or render via a small server-side wrapper. Verify in task 4 before deciding.
- **`SocialLinksEditor` reusability.** The existing artist-profile editor may bind to form-name conventions that don't match a list-of-guests context. If reuse is awkward, fall back to a small inline input pair per social link.
- **Orphan-image cleanup race.** Two organizers editing in parallel could each save with different image sets; the second save's diff would compute against the first's persisted state and could delete an image the first save still references. Acceptable risk (small org teams), but worth a comment on the action.
- **Validation error mapping.** Zod returns flat error paths; mapping them to `items.0.title`-style keys for inline display needs care. The programme editor task is the first time we'll do this — if it gets gnarly, we can simplify to a single top-of-form error list.
- **Bio length / markdown abuse.** 4 000-char cap is tight enough to prevent accidental novels but doesn't sanitise the markdown for things like inline scripts. The existing `<Markdown>` component's sanitiser is the trust boundary; verify it strips dangerous tags before relying on it for organizer-controlled input.
